STM32 安全机制概览

发表于 2019-07-27 更新于 2019-09-07 分类于 安全

安全目标

需要保护的资产:

安全系统组成

为了对抗多种攻击，数种安全机制必须在系统内实现，全面的安全需要从硬件到应用程序通篇考虑。一方面，如果设备没有对外部入侵（如调试接口）的保护，敏感数据的应用就无法被可信的实现于硬件平台上。另一方面，健壮的设备保护机制在潜在的应用漏洞（如允许访问设备全部地址空间）面前是无效的。

完整的设备保护机制包括数种存储保护机制，使其免于外部接口，控制启动模式和设备环境监视。典型的安全应用包括：

  • 安全启动，保证启动时应用完整可靠未篡改
  • 支持安全的通信协议（如 TLS）
  • 安全的固件更新能力，以安全的方式下载新的固件版本
  • 安全密钥存储（Secure Key Storage，SKS）能力

攻击类型

对微控制器的攻击分类如下：

  • 软件攻击：利用软件弱点（如协议弱点）
  • 硬件非侵入攻击：专注于利用微控制器接口和环境信息
  • 硬件侵入攻击：破坏性的接触硅片的攻击方式

软件攻击

软件攻击由恶意软件实施，恶意软件尝试获得设备控制权以访问设备的任何资源（如ID，RAM&Flash 内容，外设寄存器等）或者修改设备功能。恶意软件可以注入设备或者已经存在于应用中（如通过未经验证的库）。恶意软件有许多种，例如：

  • 访问并修改设备配置（option bytes, memory attributes…）
  • 关闭保护
  • 读取内存并导出内容，以复制数据或固件
  • 跟踪或记录设备数据
  • 访问加密数据
  • 打开通信信道/接口
  • 修改或屏蔽设备功能

除非用户应用是完全可信、无错且孤立，不使用任何方法于外部世界通信，否则软件攻击都是需要考量的。

恶意软件注入

恶意软件的尺寸取决于黑客的目标，在大多数情况下很小（几十个字节）。恶意软件需要被注入到内存或Flash中，一旦注入成功，下一步就是构建分支执行恶意软件的情况。

恶意软件注入的方法分为四类：

  1. 基本设备访问/“open doors”
    1. 调试接口：JTAG 或 SWD 接口
    2. Bootloader：如果 bootloader 可访问，可以通过任意接口读写存储
    3. 外部存储执行
  2. 应用下载
    1. 劫持固件升级过程：使恶意软件替代新固件
    2. 通过操作系统下载新应用的能力安装恶意软件
  3. 通信端口的弱点与漏洞利用
    1. 通过应用已知漏洞（溢出，缺乏数据验证）输入数据并执行
    2. 基于栈的缓冲区溢出，基于堆的缓冲区溢出，跳转到libc攻击，仅数据攻击
  4. 使用有后门的不可信库

第一类注入可以借助硬件防护解决；第二类注入可以引入基于密码的验证机制解决；第三类相对难以避免，开发嵌入式系统的低层次语言（如C/C++）常常导致内存管理错误可被攻击者利用，通常的解决方案是减少攻击面（固件中不可信/未验证的部分）；第四类往往通过网络共享的固件包添加木马实现，也需要尽可能的孤立进程执行，并保护关键代码和数据。

暴力攻击

主要用于侵入密码保护的应用，如会话验证（session authentication）和人机接口（human-machine interface），通过自动穷尽的尝试破解密码

防护：

  • 限制登入尝试次数（单调递增计数，可用 Timer 实现）
  • 增加两次登陆尝试的时间间隔
  • 增加 challenge-response 机制破坏自动尝试的条件

硬件攻击

硬件攻击需要物理接触设备。非侵入式攻击仅仅需要板级访问，而侵入式攻击需要去除封装直接访问硅片。对于侵入式攻击需要更高层次的保护，符合最新安全标准的认证需要专用硬件。

非侵入式攻击

非侵入式/板级攻击技术尝试在不物理破坏设备的前提下检索嵌入设备的信息(代码、密钥、可信信息…)，仅仅通过可访问的接口和设备环境。

调试接口

通过调试接口的攻击是最基础的形式。关闭调试功能是首先被考量的保护级别。事实上，访问调试接口或JTAG/SWD的扫描链允许访问全部的设备内部资源：CPU寄存器，片上 Flash，RAM 和外设寄存器

反制措施：

  • 关闭调试接口或通过Readout protection(RDP)熔断

串行接口

I2C、SPI 等通信接口的访问能力也可能被利用。通信接口可以看作一个设备访问入口。攻击者可以访问的资源依赖于相关协议实现（存储地址访问范围，目标外设，读写操作等）。

反制措施：

  • 软件：
    • 相关协议操作应当被限制在固件级，不能读写敏感资源
    • 隔离通信栈和敏感数据
    • 数据传输长度需要被控制在不会造成缓存溢出的长度
    • 通信可以被本设备和目标设备的共享密钥加密
  • 硬件：
    • 物理通信接口可以埋在多层板中，使攻击者难以访问
    • 关闭不使用的通信结偶

错误注入：时钟/供电干扰毛刺攻击

错误注入，通过在异常条件下使用设备，使系统进入异常状态。成功的错误注入攻击会以不同的方式修改程序的行为导致严重的后果：

  • 破坏程序状态
  • 破坏内存内容
  • 停止进程执行（”stuck-at fault”）
  • 跳过指令
  • 修改条件跳转
  • 提供未授权的访问

典型的攻击方式包括时钟（冻结、尖刺）和供电（欠压、过压、尖刺）。由于错误注入可能是无意为之，反制措施措施与安全措施相同：冗余，错误检测和监控。

反制措施：

  • 软件：
    • 冗余（或重计算）以加固敏感代码段条件跳转
  • 硬件：
    • 使用时钟安全系统（clock security system）
    • 使用内部时钟源
    • 使用内部稳压器
    • 使用存储错误检测（ECC/parity）

边信道攻击

当固件执行时，攻击者可以观察设备的模拟特性：功耗、电磁辐射、温度、活跃时间等。观测这些物理特性可以带来充分的信息来获得秘密资产如数据、算法实现等。边信道攻击对破解加密设备密钥而言非常有效。Simple Power Analysis 和 Differential Power Analysis 是典型的利用功耗进行边信道攻击的手段。

反制措施：

  • 软件：
    • 限制密钥使用：在可能时，使用会话随机密钥
    • 使用受保护的密码学库，包括随机行为（延迟、伪指令等）
  • 硬件：
    • 在安全组件中有反监测防护，在通用微控制器中没有这样高效的反制措施

侵入式攻击

进行侵入式攻击成熟的成本高昂，同时也接受使用任何方法而不在意在过程中设备被摧毁。侵入式攻击首先需要起初设备封装。一些分析技术可以不去除表面钝化层，然而想通过探针与设备交互就必须去除钝化层了。去封装可以通过化学蚀刻、钻孔、激光切割实现。

逆向工程

目标是理解设备内部结构、学习其功能。第一步是建立一个微控制器映射，可以使用光学显微镜产生高分辨率设备表面图像，接下来需要进一步蚀刻，以对更深的层次拍照。

微探针和内部错误注入

微探针与设备的金属层交互。一些薄电极用来和引脚连接，使得攻击者可以在设备运行时观察、操作、干涉它。

设备修改

使用 Focused Ion Beam 工作站可以简化探针对深层金属和多晶硅的手动操作，甚至可以通过改变内部连接线来修改设备结构。

典型物联网系统威胁

IoT System

一个完整的物联网系统包括的所有软硬件组件，如上图所示。潜在的攻击目标包括：

1. 初始准备 Initial Provisioning

一些系统依赖于基础的敏感数据来建立进一步的安全应用（安全启动，安全固件更新）。这些例如密钥（作为信任根基）的敏感数据，应该保持私密，例如以哈希值、证书的方式保存。敏感数据可以划分单独的安全存储区。初始准备应该在可信环境下完成，使用 STM32 安全服务，如 Security Firmware Install（SFI）。一旦设备内部被编程，数据保护机制应该被打开，且仅授权的进程能反问。

  • 风险：固件损坏，篡改
  • 反制措施：
    • 受信任的制造商签名
    • 使用案前的数据准备服务（SFI）
    • 数据保护机制
    • 安全应用隔离

2. 启动修改 Boot Modification

这种类型的攻击使用 bootloader 作为访问设备的入口。通过修改启动模式、地址，抢占CPU。抢占的手法有：调试接口和内存注入固件，启动模式和地址由设备配置项/输入管脚控制。

  • 风险：完全访问微控制器内容
  • 反制措施：
    • 单独的启动入口
    • 关闭 bootloader 和 debug - Readout protection(RDP)

3. 安全启动 Secure Boot

系统健壮性基于于启动主应用前对固件完整性和可靠性检查，安全启动应用在启动时执行，完成上述检查。安全启动作为设备级信任的基础，执行安全启动的这部分固件应该不可修改和绕过。

成功的对安全启动的攻击可以通过错误注入直接跳转到恶意软件或在初始准备阶段通过替换目标hash值绕过验证，执行不受信任的应用。

  • 风险：设备欺骗，应用修改等
  • 反制措施：
    • 单独的启动入口以避免验证绕过
    • 令安全启动代码不可修改
    • 对固件签名使用安全存储
    • 环境事件检测（glitch, temp, clock, …）

4. 固件更新 Firmware Update

固件更新（升级）过程允许设备所有者选择正确的固件版本以提供最好的用户体验。然而，固件更新是个脆弱的过程，会给攻击者将自己的固件或修改过的已有固件注入设备的机会。

这一过程需要通过固件完整性和可靠性检验来保证安全。成功的攻击需要访问密钥和加密过程，主要的威胁在初始准备阶段。

  • 风险：设备固件篡改
  • 反制措施：具有完整性和可靠性检查的安全固件更新应用，进一步的安全性可以在签名之上进一步加密固件。

5. 通信接口 Communication Interfaces

串行通信接口（SPI, I2C, USART…）常用于 bootloader、用户应用与设备交换数据和指令。物理访问通信接口使攻击者可以窃听并使用该接口作为设备入口。固件中通信协议也易于产生漏洞（溢出等）

6. 调试接口 Debug port

调试接口提供访问设备的全部内容：核心、外设寄存器、Flash和SRAM。用于应用开发，

  • 风险：完全访问设备
  • 反制措施：关闭设备调试能力（Readout protection）

7. 外设访问 External peripheral access

一个物联网设备控制传感器和执行器依赖于全局应用。一个攻击者通过修改或转移系统从传感器获得的数据以及分流输出给执行器的数据。

  • 风险：阻碍正确的系统行为
  • 反制措施：防篡改以在板级测系统侵入

8. 敏感固件和数据 Sensitive firmware and data

固件中的一部分需要特殊保护：例如最敏感的算法或第三方库。此外，一些被考虑为资产的数据（如密钥）需要增强保护。内部存储内容应该避免外部访问（调试接口，通信接口等）和内部访问（其他软件进程）。存储特性和进程资源防火墙是主要的保护措施来进行数据和进程隔离。

  • 风险：敏感固件复制，数据盗窃
  • 反制措施：仅执行访问权（XO），防火墙，存储保护单元，安全区…

9. 内存 SRAM

SRAM是设备的运行内存。它含有运行时缓冲区和变量（栈、堆…）、固件和密钥。SRAM的保护力度也许不如Flash那么完备，其内容为运行时产生，难以控制。这使得攻击者开发了三种主流攻击方式：代码（恶意软件）注入，内存篡改（通过缓存溢出）和检索资产（通过临时变量）

  • 风险：缓存溢出，数据偷取，设备控制
  • 反制措施：存储保护单元，保护RAM使用

10. 随机数生成 Random number generation

随机数常常用于密码学生成会话密钥或初始向量。事实上，通信的保密性依赖于密钥的健壮性。脆弱的随机数生成器会使安全协议易于攻击。如利用某种隐藏的周期性或随机序列结构来猜测密钥，从而打破通信的保密性。健壮的随机数发生器依赖于熵源的质量（模拟电路）。

  • 风险：降低密码协议安全性
  • 反制措施：使用珍重的硬件熵发生器

11. 通信栈 Communication stack

连接协议（蓝牙，以太网，Wi-Fi，LoRa…）需要复杂的通信固件栈支持。这些协议栈通常是开源的，不应认为是可信且最优的。这些开源协议栈的弱点可以大规模的被利用。

  • 风险：网络访问设备（内容，控制）
  • 反制措施：通信进程隔离，服务器授权

12. 窃听通信 Communication eavesdrop

设备和物联网服务间数据交换可以被直接通过适当的射频设备或者网络监听。一个黑客会在其中检索数据，得到设备ID，访问服务等。

密码学被所有协议接受为防窃听手段。在不同层次（设备、网管、应用）保护通信

  • 风险：观察和嗅探网络流量
  • 反制措施：使用有密码的通信协议栈（TLS for Ethernet…）

硬件保护

这里提及的保护机制由硬件机制控制，由选项字节或硬件组件配置。

  • 存储保护：主要的安全特性以保护代码和数据免于内部软件和外部的攻击
  • 软件隔离：进程间保护以避免内部攻击
  • 接口保护：保护设备接入点（串口，调试口等）
  • 系统监控：检测设备外部篡改尝试或者异常行为

存储保护

存储保护是系统安全是最重要的部分。当含有敏感代码和数据时，存储必须不能被意外的接口（如调试接口）或未授权的进程访问。使用的安全机制取决于保护的资产类型（代码/数据）、避免的访问源（外部接口/内部进程）以及存储类型（Flash、SRAM和外部存储）。部分访问过滤可以配置在存储接口（Flash控制器），总线控制IP（防火墙）或处理器内存保护单元。嵌入式 Flash、SRAM 和外存组成的存储结构如图：

系统 Flash 存储

在 STM32 中，系统存储是嵌入式 Flash 的只读部分，由 ST Bootloader 使用。一些型号在这一存储域配有一些安全服务（RSS）。系统存储的不可篡改性，是保证设备固件可认证且完整的的基石。ST Bootloader，由于不包含任何敏感算法，是可读的。一些RSS保护的部分是对用户不可见也不可读的，系统 Flash 存储的保护参数也是不可修改的。

用户 Flash 存储

用户 Flash 存储主要用于存储固件和非易失性数据。作为嵌入式 Flash 的一部分，在全部 STM32 系列设备上，都具有一系列的对用户 Flash 存储的保护措施。

外部攻击

嵌入式 Flash 可以通过简单地通过提供避免调试接口访问和控制接口访问权限来隔离外部攻击。

相关保护措施：RDP 使调试访问失效。

内部攻击

内部的读写访问可能来自注入于设备内存或已经在嵌入式芯片内部的不受信任的库中，所以关键的数据和代码应该仅仅被授权的进程访问。

相关保护措施：PCROP，MPU，firewall，Secure User Area

保护未使用存储

写保护应当作为 Flash 存储的默认选项，即使在未使用的区域，以免被代码篡改和注入。一个最佳实践是填充未使用的存储为某个已知值（如软件中断操作码、不合法的操作码或者NOPs）

错误码纠正

部分 Flash 存储具有错误码检测和纠正特性（1bit，或2bit），在可用的情况下，是一个值得开启的安全选项。

嵌入式 SRAM

设备的工作内存，它包含了堆、栈、全局缓冲和运行时变量。SRAM 可以，以字节、半字、字为单位，用最大时钟频率访问而无需进入等待状态。
